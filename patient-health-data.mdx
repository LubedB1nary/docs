# Patient Health Data

MDI stores a rich set of clinical information on each patient. This data is what clinicians use to make prescribing decisions, so getting it right is essential. This guide covers every health-related field, how vitals work, how exams create a history over time, and how ICD disease codes integrate with cases.

## Health Fields on the Patient Record

When you create or update a patient, you can include clinical health data directly on the patient record. These fields persist across cases — they represent the patient's ongoing health profile.

### Vitals & Measurements

| Field | Type | Unit | Notes |
|-------|------|------|-------|
| `weight` | float (0-999) | **Kilograms** | Required for underage patients. Used in BMI calculation for conditional rules. |
| `height` | integer (0-300) | **Centimeters** | Required for underage patients. Used in BMI calculation. |
| `body_temperature` | float (0.0-100.0) | **Celsius** | Optional. Collected if questionnaire has `is_body_temperature_asked: true`. |
| `oxygen_saturation` | integer (0-100) | **Percentage** | Optional. Collected if questionnaire has `is_oxygen_saturation_asked: true`. |
| `blood_pressure` | string | — | Format like "120-129" or "120-80". Collected if questionnaire has `ask_blood_pressure: true`. |

<Warning>
  Weight is in **kilograms** and height is in **centimeters** — not pounds and inches. If your app collects data in imperial units, you'll need to convert before sending to MDI. Common conversions: 1 lb = 0.4536 kg, 1 inch = 2.54 cm.
</Warning>

### BMI and Conditional Rules

MDI calculates BMI automatically from the patient's height and weight. You don't send BMI directly — it's derived. This calculated BMI is used by the questionnaire rules engine to conditionally show questions or include offerings.

For example, a weight loss program might have a rule that only includes GLP-1 medications (like Semaglutide) if the patient's BMI is above 27:

```json
{
  "based_on": "bmi",
  "conditional_answer": ">=",
  "required_answer": "27"
}
```

This is why getting accurate weight and height data matters — it determines what medications the patient is eligible for.

### Medical History

| Field | Type | Description |
|-------|------|-------------|
| `allergies` | text | Known drug allergies and intolerances. Free text. Examples: "Penicillin", "Sulfa drugs, peanuts". |
| `current_medications` | text | Medications the patient is currently taking. Examples: "Metformin 500mg twice daily, Lisinopril 10mg daily". |
| `medical_conditions` | text | Known medical conditions. Examples: "Type 2 diabetes, hypertension". |
| `special_necessities` | text | Any special needs or notes. Examples: "Patient is hard of hearing", "Requires Spanish-language materials". |
| `pregnancy` | boolean | Whether the patient is currently pregnant or expecting. Critical for teratogenic medications. |

<Note>
  These fields are free text, not structured data. The clinician reads them as part of their case review. If you're collecting this information through your own intake form, include as much detail as the patient provides — medication names, dosages, and frequencies are all helpful.
</Note>

## Patient Exams — Health Snapshots Over Time

Exams are point-in-time snapshots of a patient's health vitals. Each time a patient goes through an intake or a clinician requests updated health information, a new exam record is created. This gives you a longitudinal view of the patient's health data.

### What's in an Exam

```json
{
  "id": "8cc8e948-1578-409b-9e18-0a771d349355",
  "patient_id": "5c791d01-2ae5-4e44-bf4f-900dfc1ba325",
  "height": 165,
  "weight": 88.5,
  "medications": "Metformin 500mg twice daily",
  "medical_conditions": "Type 2 diabetes",
  "allergies": "Penicillin",
  "created_at": "2023-12-05T14:34:59.000000Z",
  "updated_at": "2023-12-05T14:34:59.000000Z"
}
```

### List All Exams for a Patient

<CodeGroup>

```bash cURL
curl -X GET https://api.mdintegrations.com/v1/partner/patients/PATIENT_ID/exams \
  -H "Accept: application/json" \
  -H "Authorization: Bearer TOKEN"
```

```python Python
import requests

exams = requests.get(
    f"https://api.mdintegrations.com/v1/partner/patients/{patient_id}/exams",
    headers={"Authorization": "Bearer TOKEN", "Accept": "application/json"}
).json()

for exam in exams.get("data", []):
    bmi = None
    if exam.get("weight") and exam.get("height"):
        height_m = exam["height"] / 100
        bmi = round(exam["weight"] / (height_m ** 2), 1)
    print(f"Exam {exam['id']} ({exam['created_at'][:10]})")
    print(f"  Weight: {exam.get('weight')} kg, Height: {exam.get('height')} cm, BMI: {bmi}")
    print(f"  Allergies: {exam.get('allergies', 'None reported')}")
    print(f"  Medications: {exam.get('medications', 'None reported')}")
    print(f"  Conditions: {exam.get('medical_conditions', 'None reported')}")
```

```javascript JavaScript
const response = await fetch(
  `https://api.mdintegrations.com/v1/partner/patients/${patientId}/exams`,
  { headers: { "Authorization": "Bearer TOKEN", "Accept": "application/json" } }
);
const { data: exams } = await response.json();
exams.forEach(exam => {
  const bmi = exam.weight && exam.height
    ? (exam.weight / Math.pow(exam.height / 100, 2)).toFixed(1)
    : null;
  console.log(`Exam ${exam.id}: Weight ${exam.weight}kg, Height ${exam.height}cm, BMI ${bmi}`);
});
```


### Requesting Updated Exams from Patients

When a clinician needs updated vitals (e.g., the patient's weight has changed since their last visit), they can request an exam update. This triggers a webhook to your system:

```json
{
  "timestamp": 1700768392,
  "event_type": "exam_requested",
  "access_link": "https://patient.mdintegrations.com/exam/TOKEN?fullscreen=true",
  "patient_id": "796fef8b-a4d5-4e19-8a58-852adfbe8709",
  "metadata": "This is an example of metadata field."
}
```

The `access_link` is a pre-authenticated URL you can send to the patient. They fill in their updated vitals and the exam record is created automatically.

## Disease Codes (ICD)

Disease codes identify the medical diagnosis associated with a case. They're required for e-prescribing — DoseSpot needs at least one ICD code to submit a prescription.

### Understanding ICD Codes

ICD (International Classification of Diseases) codes are standardized diagnosis codes used across healthcare. Examples:

| ICD Code | Description |
|----------|-------------|
| E66.01 | Morbid obesity due to excess calories |
| L70.0 | Acne vulgaris |
| E11.9 | Type 2 diabetes without complications |
| N94.6 | Dysmenorrhea, unspecified |

### Listing Available Disease Codes

### Attaching Diseases to a Case

You can attach diseases when creating a case (in the `diseases` array) or add them afterward:

```bash
curl -X POST https://api.mdintegrations.com/v1/partner/cases/CASE_ID/diseases \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer TOKEN" \
  -d '{"disease_id": "0b333a71-8e31-4b5d-834c-27f724ac14ff"}'
```

### Setting a Primary Disease

Each case can have multiple diseases, but one must be designated as primary (sent to DoseSpot for the prescription):

```bash
curl -X POST https://api.mdintegrations.com/v1/partner/cases/CASE_ID/diseases/DISEASE_ID/primary \
  -H "Authorization: Bearer TOKEN"
```

### Diseases on Offerings

Diseases can also be attached at the offering (prescription) level, not just the case level. This is important when a case has multiple prescriptions for different conditions:

```bash
# List diseases on a specific offering
curl -X GET https://api.mdintegrations.com/v1/partner/cases/CASE_ID/offerings/OFFERING_ID/diseases \
  -H "Authorization: Bearer TOKEN"

# Attach a disease to an offering
curl -X POST https://api.mdintegrations.com/v1/partner/cases/CASE_ID/offerings/OFFERING_ID/diseases \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer TOKEN" \
  -d '{"disease_id": "DISEASE_UUID"}'
```

## Custom Patient Metafields

For any health data that doesn't fit into the standard fields, use metafields — custom key-value pairs stored on the patient:

```json
"metafields": [
  {
    "key": "smoking_status",
    "title": "Smoking Status",
    "value": "former",
    "type": "string"
  },
  {
    "key": "exercise_frequency",
    "title": "How often do you exercise?",
    "value": "3-4 times per week",
    "type": "string"
  },
  {
    "key": "family_history_diabetes",
    "title": "Family history of diabetes?",
    "value": "true",
    "type": "boolean"
  }
]
```

Metafields support types: `string`, `boolean`, `number`. They're flexible enough to store any structured health data your intake collects beyond the standard MDI fields.

## Putting It All Together

Here's how health data flows through a typical integration:

1. **Patient creation** — You collect basic vitals (weight, height) and medical history (allergies, medications, conditions) and include them when creating the patient.

2. **Voucher/Intake** — During intake, the questionnaire may collect additional vitals (blood pressure, body temperature, oxygen saturation) depending on its configuration. BMI is calculated from height + weight.

3. **Conditional rules fire** — Based on the patient's state, BMI, and questionnaire answers, rules determine which questions appear and which offerings are included in the case.

4. **Case creation** — Diseases are attached (either from intake rules or your direct API call). Questions with critical/important flags are highlighted for the clinician.

5. **Clinician review** — The clinician sees all health data: vitals, allergies, conditions, medications, questionnaire answers (with important/critical flags), and attached diseases.

6. **Exam record created** — A snapshot of the patient's health at this point in time is saved as an exam. Future cases can reference the exam history.

7. **Follow-up cases** — When the patient comes back, their health data carries over. The questionnaire may ask for updated vitals if `always_ask_weight_height` is enabled.
